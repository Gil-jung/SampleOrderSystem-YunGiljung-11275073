# PLAN.md — 구현 계획 (반도체 시료 생산주문관리 시스템)

본 문서는 [`PRD.md`](./PRD.md)의 기능 요구사항을 바탕으로, 앞으로 [`tdd` 스킬](./.claude/skills/tdd/SKILL.md)을
사용해 진행할 구현 순서와 각 사이클의 목표를 정리한 전체 로드맵이다.

> **문서 성격**: 이 PLAN.md는 "전체 로드맵"이다. tdd 스킬의 RED 단계에서 요구하는
> "사이클별 PLAN.md"(목표 1개 + 실패 테스트 + 구현 방향)는 아래 각 사이클을 시작할 때
> 별도로 작성/갱신하며, 이 문서의 해당 사이클 절을 근거로 삼는다. 즉 이 문서는
> "무엇을 어떤 순서로 만들 것인가"의 지도이고, 사이클별 PLAN.md는 "지금 이 한 걸음을
> 어떻게 검증할 것인가"의 실행 문서다.

## 1. 아키텍처 개요

`PoC/ConsoleMVC`에서 검증한 계층 분리 원칙을 그대로 따르되, 비즈니스 규칙(재고 분기,
생산 큐, 수율 계산)이 늘어나는 만큼 Model과 Controller 사이에 **Service** 계층과
**Repository(저장소)** 계층을 추가한다.

```
Controller → Service → Repository
Controller → View
Service → Model
```

| 계층 | 책임 | 하지 않는 일 |
|---|---|---|
| **Model** | Sample/Order 데이터와 스스로 보장해야 하는 불변식(수량 > 0, 상태 전이 유효성 등) | 저장/조회, 유스케이스 조율, 출력 |
| **Repository** | Sample/Order를 인메모리(dict/list)에 저장하고 조회하는 최소 CRUD | 비즈니스 규칙 판단 |
| **Service** | 유스케이스별 비즈니스 규칙(재고 분기, FIFO 생산 큐, 수율 기반 실생산량 계산, 상태 전이) | 콘솔 입출력, 저장 방식의 구체 구현 |
| **Controller** | 메뉴 입력을 받아 Service를 호출하고, 결과를 View가 그릴 수 있는 `dict`로 변환 | 비즈니스 규칙 자체 구현 |
| **View** | Controller가 준 `dict`를 사람이 읽을 문자열로 렌더링 | 상태 변경, Model/Service 참조 |

## 2. 디렉터리 구조 (제안)

```
src/
├── model/
│   ├── sample.py            # Sample (값 객체), 수율 계산 헬퍼
│   └── order.py             # Order, OrderStatus, OrderError(상태 전이 예외)
├── repository/
│   ├── sample_repository.py # SampleRepository: add/get/list/search
│   └── order_repository.py  # OrderRepository: add/get/list(status별)
├── service/
│   ├── sample_service.py     # 시료 등록/조회/검색 유스케이스
│   ├── order_service.py      # 주문 예약/승인/거절 유스케이스 (재고 분기 포함)
│   ├── production_service.py # 생산 큐(FIFO), 실생산량/총생산시간 계산, 생산 완료 처리
│   ├── monitoring_service.py # 상태별 주문 집계, 재고 상태(여유/부족/고갈) 판정
│   └── release_service.py    # 출고 처리 (CONFIRMED -> RELEASE)
├── controller/
│   └── menu_controller.py    # 메인 메뉴 각 항목의 유스케이스 조율
├── view/
│   └── console_view.py       # 메뉴/목록/요약 렌더링
├── main.py                    # 콘솔 진입점 (메뉴 루프)
tests/
├── model/
├── repository/
├── service/
└── controller/  (필요 시)
```

기존 `PoC/`는 변경하지 않고, 신규 구현은 저장소 루트의 `src/`, `tests/`에 작성한다.

## 3. 공통 원칙

- 모든 사이클은 `tdd` 스킬의 Red → Green(→ Refactor) 순서를 따르고, RED 단계에서 사람 파트너의
  승인 없이는 구현 코드를 작성하지 않는다.
- 저장소는 PoC와 동일하게 **인메모리**(dict/list)로 시작한다. 영속화(JSON 파일 등)는 PRD 범위 밖이며,
  필요 시 별도 논의 후 Repository 구현체만 교체한다.
- 하나의 사이클 = 하나의 관찰 가능한 동작(behavior). 아래 사이클 목록의 각 항목이 곧
  하나의 RED-GREEN 단위이며, 필요하면 더 잘게 쪼갤 수 있다(합치지는 않는다).
- 수량/수율 등 입력값 검증(0 이하 수량 거부, 수율 범위 0~1 등)은 별도 사이클로 분리하여
  "정상 케이스"와 "예외 케이스" 테스트를 각각 명확히 작성한다.

## 4. 마일스톤 (Phase) 개요

| Phase | 내용 | PRD 참조 |
|---|---|---|
| 0 | 프로젝트 스캐폴딩 (`src/`, `tests/`, pytest 설정) | - |
| 1 | Model: Sample / Order / OrderStatus | 2절 |
| 2 | 시료 관리 (등록/조회/검색) | 4.1절 |
| 3 | 시료 주문(예약) | 4.2절 |
| 4 | 주문 승인/거절 (재고 분기 포함) | 4.3절 |
| 5 | 생산 라인 (FIFO 큐, 수율 기반 계산, 생산 완료) | 4.5절 |
| 6 | 모니터링 (주문량/재고량) | 4.4절 |
| 7 | 출고 처리 | 4.6절 |
| 8 | 콘솔 통합 (Controller/View/main, 메인 메뉴) | 3절 |
| 9 | 종합 시나리오 검증 (E2E) | 전체 |

Phase 4(승인/거절)가 Phase 5(생산 라인)보다 먼저지만, 실제로는 두 Phase가 강하게 얽혀 있다
(승인 시 재고 부족 분기 → 생산 큐 등록 → 생산 완료 시 주문 상태 전환). 아래 사이클 목록에서는
이 의존관계를 고려해 순서를 정했다.

## 5. 사이클별 상세 계획

각 사이클은 `[Phase-번호]` 형식으로 표기한다. "핵심 테스트"는 RED 단계에서 작성할 대표
테스트 이름/의도이며, 실제 작성 시 필요하면 더 세분화한다.

### Phase 0. 스캐폴딩

- **[0-1] pytest 실행 환경 구성**: `pytest.ini` 또는 `pyproject.toml`로 `src`/`tests` 경로 인식 확인.
  테스트 코드가 없는 상태이므로 TDD 대상이 아니라 설정 작업으로 간주하고, 완료 후 간단한
  placeholder 테스트(`test_pytest_discovers_tests`) 하나로 스캐폴딩 동작을 확인한다.

### Phase 1. Model — Sample / Order

- **[1-1] Sample 값 객체 생성**: `Sample(sample_id, name, avg_production_time, yield_rate)` 생성 시
  필드가 그대로 보존된다. 초기 `stock`은 0.
- **[1-2] Sample 수율 범위 검증**: `yield_rate`가 0~1 범위를 벗어나면 `ValueError`(또는 전용 예외).
- **[1-3] Order 값 객체 생성 및 초기 상태**: `Order(sample_id, customer_name, quantity)` 생성 시
  상태는 `RESERVED`.
- **[1-4] Order 수량 검증**: `quantity <= 0`이면 생성 거부(예외).
- **[1-5] OrderStatus 전이 규칙 (+ [1-6] 종결 상태 보호 통합)**: 허용된 전이만 성공.
  - `RESERVED → CONFIRMED`, `RESERVED → PRODUCING`, `RESERVED → REJECTED`,
    `PRODUCING → CONFIRMED`, `CONFIRMED → RELEASE` 는 성공
  - 그 외(예: `RELEASE → CONFIRMED`, `REJECTED → CONFIRMED`)는 `OrderError`로 거부
  - **[1-6]과 통합한 이유** (`docs/1-5.md` 참조): 종결 상태(`REJECTED`/`RELEASE`)의 전이 불가는
    동일한 허용 전이 테이블에서 해당 상태의 허용 목록이 비어있다는 사실의 자연스러운 결과이다.
    [1-5]를 먼저 구현하면 [1-6] 테스트는 이미 통과된 상태가 되어 RED(실패)를 볼 수 없으므로,
    두 사이클을 하나로 합쳐 유효 전이 5종 + 종결 상태 포함 무효 전이 다수를 함께 검증했다.

### Phase 2. 시료 관리

- **[2-1] 시료 등록**: `SampleService.register(sample_id, name, avg_production_time, yield_rate)` 호출 시
  `SampleRepository`에 저장되고 재고 0으로 시작.
- **[2-2] 중복 ID 등록 거부**: 이미 존재하는 `sample_id`로 등록 시 예외.
- **[2-3] 전체 시료 조회**: `SampleService.list_all()`이 등록된 모든 시료와 현재 재고를 함께 반환.
- **[2-4] 이름으로 시료 검색**: `SampleService.search(name)`이 부분 일치하는 시료만 반환.
- **[2-5] 빈 검색 결과**: 일치하는 시료가 없으면 빈 목록 반환(예외 아님).
  - **RED 없이 확인만 진행함** (`docs/2-5.md` 참조): [2-4]의 리스트 컴프리헨션 구현이 일치하는
    항목이 없을 때 이미 빈 리스트를 반환하므로, 이 사이클은 새 프로덕션 코드 없이 회귀
    방지용 확인 테스트만 추가했다.

### Phase 3. 시료 주문 (예약)

- **[3-1] 주문 예약 생성**: `OrderService.reserve(sample_id, customer_name, quantity)` 호출 시
  `RESERVED` 상태 주문이 생성되고 `OrderRepository`에 저장됨.
- **[3-2] 존재하지 않는 시료로 예약 거부**: 등록되지 않은 `sample_id`면 예외.
- **[3-3] 0 이하 수량 예약 거부**: Model의 [1-4] 규칙이 Service를 통해서도 지켜짐(통합 확인).
  - **RED 없이 확인만 진행함** (`docs/3-3.md` 참조): `OrderService.reserve`가 `Order(...)`
    생성자를 그대로 호출하므로 [1-4] + [3-2] 구현만으로 이미 충족되어, 새 프로덕션 코드
    없이 회귀 방지용 확인 테스트만 추가했다.

### Phase 4. 주문 승인/거절

- **[4-1] 접수된 주문 목록 조회**: `OrderService.list_reserved()`가 `RESERVED` 상태 주문만 반환.
- **[4-2] 승인 — 재고 충분**: 시료 재고 ≥ 주문 수량인 `RESERVED` 주문을 승인하면
  - 주문 상태가 `CONFIRMED`로 전환
  - 시료 재고가 주문 수량만큼 즉시 차감(예약분 반영)
- **[4-3] 승인 — 재고 부족**: 재고 < 주문 수량인 경우
  - 주문 상태가 `PRODUCING`으로 전환
  - 생산 큐(Phase 5에서 구현할 `ProductionService`)에 해당 주문이 등록됨
  - (이 사이클에서는 "큐에 등록되었다"까지만 검증하고, 큐 내부 계산은 Phase 5로 넘긴다)
- **[4-4] 존재하지 않는 주문 승인 시도**: 예외.
- **[4-5] 이미 처리된(RESERVED가 아닌) 주문 승인 시도 거부**: `OrderError`.
  - **RED 없이 확인만 진행함** (`docs/4-5.md` 참조): `approve`가 재고 분기 어느 쪽이든
    `order.transition_to(...)`를 호출하므로, [1-5]의 전이 테이블만으로 이미 재전이가
    `OrderError`로 거부되어 새 프로덕션 코드 없이 확인 테스트만 추가했다.
- **[4-6] 주문 거절**: `RESERVED` 주문을 거절하면 즉시 `REJECTED`로 전환, 재고 변화 없음.
- **[4-7] 이미 처리된 주문 거절 시도 거부**: `OrderError`.
  - **RED 없이 확인만 진행함** (`docs/4-7.md` 참조): `reject`가 `order.transition_to(REJECTED)`만
    호출하는데, [1-5]의 전이 테이블에서 `REJECTED`로 갈 수 있는 출발 상태는 `RESERVED`뿐이므로
    이미 처리된 주문의 거절 시도는 이미 `OrderError`로 거부되어 새 프로덕션 코드 없이
    확인 테스트만 추가했다.

### Phase 5. 생산 라인

- **[5-1] 부족분 계산**: 승인 시점 가용 재고와 주문 수량으로부터
  `부족분 = 주문 수량 - 승인 시점 가용 재고` 계산 함수 단위 검증.
- **[5-2] 실 생산량 계산 (수율 반영)**: `ceil(부족분 / 수율)` 계산.
  - 예: 부족분 9, 수율 0.9 → 실 생산량 10
  - 나누어떨어지지 않는 경우 올림 확인 (예: 부족분 5, 수율 0.9 → `ceil(5/0.9)=6`)
- **[5-3] 총 생산 시간 계산**: `평균 생산시간 × 실 생산량`.
- **[5-4] 생산 큐 등록 — FIFO 순서 보장**: 여러 건이 `PRODUCING`으로 전환될 때
  등록 순서대로 큐에 쌓이고, 먼저 등록된 주문이 먼저 처리 대상이 됨.
  - **RED 없이 확인만 진행함** (`docs/5-4.md` 참조): [4-3]의 `ProductionService`(`list.append`
    + 순서 그대로 반환)만으로 이미 FIFO 순서가 보장되어, 새 프로덕션 코드 없이 확인
    테스트만 추가했다.
- **[5-5] 대기 주문 목록 조회**: `ProductionService.list_queue()`가 큐에 남은 주문을
  FIFO 순서 그대로 반환.
  - **[5-4]로 이미 완전히 검증됨** (`docs/5-5.md` 참조): [5-4]의 확인 테스트가
    `list_queue()`의 반환 순서 자체를 검증하고 있어(큐 순서는 `list_queue()`를 통해서만
    관찰 가능), 이 사이클은 별도 테스트나 프로덕션 코드 변경 없이 문서 메모로만 종결한다.
- **[5-6] 생산 진행 상황 조회**: 현재 생산 중인 항목(주문, 목표 실생산량, 총 생산 시간 등)을
  `ProductionService.current_status()`로 조회 가능.
- **[5-7] 생산 완료 처리**: 생산이 완료되면
  - 정상 생산분(주문 수량을 채우는 만큼)이 시료 재고에 반영
  - 해당 주문 상태가 `PRODUCING → CONFIRMED`로 전환
  - 큐에서 다음 대기 주문으로 진행

> 생산 "진행 시간"을 실시간으로 흘려보낼지, 콘솔 명령(예: "생산 완료 처리")으로 즉시
> 완료 처리할지는 구현 시 단순한 쪽(즉시완료 트리거 방식)으로 시작하고, 필요 시 사람
> 파트너와 논의 후 확장한다.

### Phase 6. 모니터링

- **[6-1] 상태별 주문 수 집계**: `RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE` 각각의 건수를 집계.
- **[6-2] REJECTED 제외 확인**: `REJECTED` 주문은 집계에 포함되지 않음.
  - **RED 없이 확인만 진행함** (`docs/6-2.md` 참조): [6-1]의 `count_by_status()`가
    `REJECTED`를 애초에 집계 키로 두지 않으므로 이미 제외되어, 새 프로덕션 코드 없이
    확인 테스트만 추가했다.
- **[6-3] 시료별 재고 상태 판정 — 여유**: 재고 ≥ 해당 시료의 미출고(`CONFIRMED`) 수요 합.
- **[6-4] 시료별 재고 상태 판정 — 부족**: 0 < 재고 < 미출고 수요 합.
- **[6-5] 시료별 재고 상태 판정 — 고갈**: 재고 == 0.
- **[6-6] 미출고 수요가 없는 시료**: 재고와 무관하게 "여유"로 판정(0 ≥ 0 규칙 확인용 경계 케이스).
  - **RED 없이 확인만 진행함** (`docs/6-6.md` 참조): `demand`가 0이면 `stock >= demand`가
    항상 참이라 [6-3]~[6-5]의 `if/elif/else` 구조만으로 이미 충족되어, 새 프로덕션 코드
    없이 확인 테스트만 추가했다.

### Phase 7. 출고 처리

- **[7-1] CONFIRMED 주문 출고**: `ReleaseService.release(order_id)` 호출 시 상태가 `RELEASE`로 전환.
- **[7-2] CONFIRMED가 아닌 주문 출고 시도 거부**: `OrderError`.
  - **RED 없이 확인만 진행함** (`docs/7-2.md` 참조): `release`가 `order.transition_to(RELEASE)`만
    호출하는데, [1-5]의 전이 테이블에서 `RELEASE`로 갈 수 있는 출발 상태는 `CONFIRMED`뿐이므로
    이미 `OrderError`로 거부되어 새 프로덕션 코드 없이 확인 테스트만 추가했다.
- **[7-3] 존재하지 않는 주문 출고 시도**: 예외.

### Phase 8. 콘솔 통합

- **[8-1] 메인 메뉴 렌더링**: 5개 메뉴 항목과 전체 시료 요약(등록 수, 재고 합계 등)이 표시됨
  (View 단위 테스트 — Controller가 준 `dict`만으로 렌더링).
- **[8-2] 시료 관리 메뉴 흐름**: 등록/조회/검색 각 명령이 Controller → Service로 올바르게 위임됨.
- **[8-3] 주문 메뉴 흐름**: 예약/승인/거절 명령이 Controller → Service로 위임됨.
- **[8-4] 모니터링 메뉴 흐름**: 주문량/재고량 조회 명령이 올바른 데이터를 View에 전달.
- **[8-5] 생산 라인 메뉴 흐름**: 생산 현황/대기 큐 조회가 올바르게 표시됨.
- **[8-6] 출고 메뉴 흐름**: 출고 명령이 올바르게 위임됨.
- **[8-7] main.py 진입점**: 메뉴 루프가 정상 종료 명령까지 동작(스모크 테스트 수준).

### Phase 9. 종합 시나리오 (E2E)

`PoC/ConsoleMVC`의 `test_order_scenario.py` 패턴을 참고하여, 아래와 같은 통합 시나리오를
pytest로 작성한다 (각각 별도 사이클로 진행).

- **[9-1] 재고 충분 시나리오**: 시료 등록(재고 임의 확보) → 예약 → 승인(즉시 CONFIRMED) → 출고(RELEASE).
  - **RED 없이 확인만 진행함** (`docs/9-1.md` 참조): 등록/예약/승인/출고 각 조각이
    [4-2]/[7-1]/[8-2]/[8-3]/[8-6]에서 이미 검증되어 있어(특히 [8-6]이 예약→승인→출고
    체인 자체를 검증함), 새 프로덕션 코드 없이 E2E 확인 테스트만 추가했다.
- **[9-2] 재고 부족 시나리오**: 시료 등록(재고 0) → 예약 → 승인(PRODUCING, 생산 큐 등록) →
  생산 완료(CONFIRMED) → 출고(RELEASE).
  - **RED 없이 확인만 진행함** (`docs/9-2.md` 참조): 각 단계가 [4-3]/[5-7]/[7-1]/[8-6]에서
    이미 검증되어 있어, 새 프로덕션 코드 없이 E2E 확인 테스트만 추가했다.
    `complete_production()`은 콘솔 메뉴에 없어 테스트에서 `OrderService`를 직접 호출한다.
- **[9-3] 거절 시나리오**: 예약 → 거절(REJECTED) → 모니터링 집계에서 제외됨을 확인.
  - **RED 없이 확인만 진행함** (`docs/9-3.md` 참조): [6-2]가 동일한 흐름(Service 직접
    사용)을 이미 검증해, MenuController 경유로 재조립해도 새 프로덕션 코드 없이
    통과해 확인 테스트만 추가했다.
- **[9-4] FIFO 다중 생산 시나리오**: 재고 부족 주문 2건 이상을 승인 → 등록 순서대로 생산 완료됨을 확인.

## 6. 계산식/규칙 재확인 (PRD 매핑)

## 7. 진행 방식

- 각 사이클을 시작하기 전, 해당 사이클 번호를 명시한 별도 PLAN.md 개정(또는 사이클 전용
  섹션)을 작성해 사람 파트너의 승인을 받는다 — `tdd` 스킬의 RED 1~2단계.
- 사이클 순서는 위 5절 순서를 기본으로 하되, 진행 중 설계상 이유로 순서를 바꿔야 하면
  그 사유를 이 문서에 기록하고 조정한다.
- Phase 완료 시마다 `PRD.md`의 해당 절 요구사항이 전부 충족되었는지 6절 매핑표로 교차 확인한다.
