# [Phase 4] 사이클 4-5 — 이미 처리된 주문 승인 시도 거부 (통합 확인)

`PLAN.md`의 Phase 4 [4-5] 항목에 대한 tdd 스킬 사이클 기록.

## RED 이전 검토 — 이 사이클도 진짜 RED를 만들 수 없다

목표: `RESERVED`가 아닌 주문(이미 승인/거절된 주문)에 `OrderService.approve`를 다시
호출하면 `OrderError`로 거부된다.

`approve`의 현재 구현을 보면, 재고 분기와 무관하게 마지막에 항상
`order.transition_to(OrderStatus.CONFIRMED)` 또는 `order.transition_to(OrderStatus.PRODUCING)`를
호출한다. `Order.transition_to`는 [1-5]에서 이미 전이 테이블(`_ALLOWED_TRANSITIONS`)로
엄격히 검증되어 있어서:

- 이미 `CONFIRMED`인 주문에 다시 `transition_to(CONFIRMED)`를 호출 → `CONFIRMED`의 허용
  목록은 `{RELEASE}`뿐이므로 `OrderError`
- 이미 `PRODUCING`인 주문에 다시 `transition_to(PRODUCING)`를 호출 → `PRODUCING`의 허용
  목록은 `{CONFIRMED}`뿐이므로 `OrderError`
- `REJECTED`/`RELEASE`(종결 상태) 주문에 대해서도 허용 목록이 비어있으므로 `OrderError`

즉 재고 분기 어느 쪽을 타든 [1-5]의 전이 테이블만으로 **이미 거부되고 있다.**
[2-5]/[3-3]과 동일한 이유로 지금 확인 테스트를 작성하면 처음부터 통과해버려
RED(실패)를 볼 수 없다.

### 제안

- 새로운 프로덕션 코드 변경은 없다.
- Service → Model 경계에서 이 방어가 실제로 전파되는지는 회귀 방지 가치가 있으므로,
  **RED 없이 확인 테스트 1개만 추가**하고 즉시 통과함을 기록한다.
- `PLAN.md`에는 [4-5]가 [1-5]의 전이 테이블만으로 이미 충족되었다는 메모를 추가한다.

### 확인 테스트 (참고용 — RED 없이 바로 통과 예정)

```python
def test_이미_승인된_주문을_다시_승인하면_OrderError로_거부된다():
    order_repository = OrderRepository()
    sample_repository = SampleRepository()
    sample_repository.add(
        Sample(sample_id="SMP-001", name="Wafer-A", avg_production_time=2.5, yield_rate=0.9)
    )
    sample_repository.get("SMP-001").stock = 10
    production_service = ProductionService()
    service = OrderService(order_repository, sample_repository, production_service)
    order_id = service.reserve(sample_id="SMP-001", customer_name="홍길동", quantity=5)
    service.approve(order_id)  # 첫 승인: CONFIRMED로 전환됨

    with pytest.raises(OrderError):
        service.approve(order_id)  # 재승인 시도
```
