# [Phase 6] 사이클 6-2 — REJECTED 제외 확인 (통합 확인)

`PLAN.md`의 Phase 6 [6-2] 항목에 대한 tdd 스킬 사이클 기록.

## RED 이전 검토 — 이 사이클도 진짜 RED를 만들 수 없다

목표: `REJECTED` 주문은 `count_by_status()` 집계에 포함되지 않는다.

[6-1]에서 구현한 `MonitoringService.count_by_status()`를 보면:

```python
_MONITORED_STATUSES = [
    OrderStatus.RESERVED,
    OrderStatus.CONFIRMED,
    OrderStatus.PRODUCING,
    OrderStatus.RELEASE,
]

def count_by_status(self):
    counts = {status: 0 for status in _MONITORED_STATUSES}
    for order in self._order_repository.list():
        if order.status in counts:
            counts[order.status] += 1
    return counts
```

`counts` 딕셔너리에는 애초에 `REJECTED` 키가 없고, `if order.status in counts` 검사로
`REJECTED` 주문은 그냥 건너뛴다. 즉 **[6-1]의 구현만으로 이미 `REJECTED`가 집계에서
제외되고 있다.** [2-5]/[3-3]/[4-5]/[4-7]/[5-4]와 동일한 이유로, 지금 확인 테스트를
작성하면 처음부터 통과해버려 RED(실패)를 볼 수 없다.

### 제안

- 새로운 프로덕션 코드 변경은 없다.
- `REJECTED` 제외는 모니터링 기능의 핵심 요구사항(PRD.md 4.4절)이므로 회귀 방지 가치가
  있다. **RED 없이 확인 테스트 1개만 추가**하고 즉시 통과함을 기록한다.
- `PLAN.md`에는 [6-2]가 [6-1] 구현만으로 이미 충족되었다는 메모를 추가한다.

### 확인 테스트 (참고용 — RED 없이 바로 통과 예정)

```python
def test_REJECTED_주문은_집계에서_제외된다():
    order_repository = OrderRepository()
    sample_repository = SampleRepository()
    sample_repository.add(
        Sample(sample_id="SMP-001", name="Wafer-A", avg_production_time=2.5, yield_rate=0.9)
    )
    order_service = OrderService(order_repository, sample_repository, ProductionService())
    monitoring_service = MonitoringService(order_repository)

    order_id = order_service.reserve(sample_id="SMP-001", customer_name="홍길동", quantity=1)
    order_service.reject(order_id)

    counts = monitoring_service.count_by_status()

    assert sum(counts.values()) == 0
    assert OrderStatus.REJECTED not in counts
```
