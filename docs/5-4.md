# [Phase 5] 사이클 5-4 — 생산 큐 등록 (FIFO 순서 보장, 통합 확인)

`PLAN.md`의 Phase 5 [5-4] 항목에 대한 tdd 스킬 사이클 기록.

## RED 이전 검토 — 이 사이클도 진짜 RED를 만들 수 없다

목표: 여러 건이 `PRODUCING`으로 전환될 때 등록 순서대로 큐에 쌓이고, 먼저 등록된
주문이 먼저 처리 대상이 된다 (FIFO).

[4-3]에서 이미 구현한 `ProductionService`를 보면:

```python
class ProductionService:
    def __init__(self):
        self._queue = []

    def enqueue(self, order_id):
        self._queue.append(order_id)

    def list_queue(self):
        return list(self._queue)
```

`list.append`는 항상 끝에 추가하고, `list_queue()`는 그 순서를 그대로 복사해 반환하므로
**여러 건을 순서대로 `enqueue`하면 이미 FIFO 순서가 보장된다.** [2-5]/[3-3]/[4-5]/[4-7]과
동일한 이유로, 지금 확인 테스트를 작성하면 처음부터 통과해버려 RED(실패)를 볼 수 없다.

### 제안

- 새로운 프로덕션 코드 변경은 없다.
- 여러 주문이 승인되어 큐에 쌓이는 흐름은 회귀 방지 가치가 있으므로, **RED 없이 확인
  테스트 1개만 추가**하고 즉시 통과함을 기록한다.
- `PLAN.md`에는 [5-4]가 [4-3]의 `ProductionService` 구현으로 이미 충족되었다는 메모를 추가한다.

### 확인 테스트 (참고용 — RED 없이 바로 통과 예정)

```python
def test_여러_주문이_승인되면_등록_순서대로_큐에_쌓인다():
    order_repository = OrderRepository()
    sample_repository = SampleRepository()
    sample_repository.add(
        Sample(sample_id="SMP-001", name="Wafer-A", avg_production_time=2.5, yield_rate=0.9)
    )
    sample_repository.get("SMP-001").stock = 0
    production_service = ProductionService()
    service = OrderService(order_repository, sample_repository, production_service)

    first_id = service.reserve(sample_id="SMP-001", customer_name="홍길동", quantity=3)
    second_id = service.reserve(sample_id="SMP-001", customer_name="김철수", quantity=2)
    service.approve(first_id)
    service.approve(second_id)

    assert production_service.list_queue() == [first_id, second_id]
```
