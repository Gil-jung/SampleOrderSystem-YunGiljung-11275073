# [Phase 5] 사이클 5-7 — 생산 완료 처리

`PLAN.md`의 Phase 5 [5-7] 항목에 대한 tdd 스킬 사이클 기록.

## RED

### 목표 (단일 동작)

생산 큐 맨 앞 주문의 생산이 완료되면:
1. 정상 생산분(주문 수량을 채우는 만큼, 즉 승인 시점의 부족분)이 시료 재고에 반영된다.
2. 해당 주문 상태가 `PRODUCING → CONFIRMED`로 전환된다.
3. 큐에서 제거되고, 다음 대기 주문으로 진행한다.

(PRD.md 4.5절)

### 설계 메모

- **재고 반영량은 "부족분"만큼**: PRD는 "정상 생산분(주문 수량을 채우는 만큼)이 재고에
  반영"이라고 명시한다. 실 생산량(`actual_production`)은 수율에 따른 불량을 감안해
  올림한 값이라 정확한 정수 개수의 "정상품"을 보장하지 않으므로, 모델을 단순화해
  "완료 시 부족분만큼만 재고에 반영된다"고 규정한다. 이는 승인 시점에 계산해 이미
  [5-6]에서 큐에 저장해 둔 `actual_production`/`total_production_time`과 같은 방식으로,
  `shortage`도 `enqueue`에 함께 저장해두고 완료 시 그대로 사용한다 (재계산하지 않음).
- **재고 반영 후 즉시 차감**: [4-2](재고 충분 분기)에서는 `CONFIRMED` 전환 시
  `sample.stock -= order.quantity`로 즉시 차감해 "출고 대기 중인 수량은 가용 재고에서
  빠진다"는 불변식을 유지했다. 생산 완료 시에도 동일한 불변식을 유지하기 위해
  `sample.stock += shortage`로 부족분을 채운 뒤 `sample.stock -= order.quantity`로
  즉시 차감한다(두 연산의 합성 효과 = 이 주문이 필요로 했던 만큼만 순수하게 소비).
- **어디에 둘 것인가**: 생산 완료는 사용자가 특정 주문을 선택하는 유스케이스가 아니라
  "현재 생산 중인 항목"에 대한 시스템 트리거이므로, `OrderService.complete_production()`
  (인자 없음)으로 둔다. `OrderService`는 이미 `order_repository`/`sample_repository`/
  `production_service`를 모두 갖고 있어 조율에 적합하다.
- `ProductionService`에 큐에서 맨 앞 항목을 꺼내는 `dequeue()`를 신규 추가한다
  (지금까지는 `enqueue`/`list_queue`/`current_status`만 있고 제거 기능이 없었다).

### 작성할 실패 테스트

`tests/service/test_order_service.py`에 아래 테스트를 추가한다.

```python
def test_생산_완료_처리_시_재고에_반영되고_CONFIRMED로_전환되며_다음_대기_주문으로_진행된다():
    order_repository = OrderRepository()
    sample_repository = SampleRepository()
    sample_repository.add(
        Sample(sample_id="SMP-001", name="Wafer-A", avg_production_time=2.5, yield_rate=0.9)
    )
    sample_repository.get("SMP-001").stock = 0
    production_service = ProductionService()
    service = OrderService(order_repository, sample_repository, production_service)
    first_id = service.reserve(sample_id="SMP-001", customer_name="홍길동", quantity=9)
    second_id = service.reserve(sample_id="SMP-001", customer_name="김철수", quantity=3)
    service.approve(first_id)
    service.approve(second_id)

    service.complete_production()

    completed_order = order_repository.get(first_id)
    assert completed_order.status == OrderStatus.CONFIRMED
    assert sample_repository.get("SMP-001").stock == 0
    assert production_service.list_queue() == [second_id]
    assert production_service.current_status()["order_id"] == second_id
```

### 구현 방향 (GREEN 예고 — RED 단계에서는 작성하지 않음)

- `ProductionService`:
  - `enqueue(order_id, actual_production=None, total_production_time=None, shortage=None)`로
    `shortage`도 함께 저장.
  - `dequeue()`: `self._queue.pop(0)`으로 맨 앞 `order_id`를 꺼내고, `self._status`에서도
    해당 항목을 제거한 뒤 `order_id`를 반환.
- `OrderService`:
  - `approve`에서 `enqueue` 호출 시 `shortage`도 함께 전달.
  - `complete_production()`:
    1. `status = self._production_service.current_status()`
    2. `order = self._order_repository.get(status["order_id"])`
    3. `sample = self._sample_repository.get(order.sample_id)`
    4. `sample.stock += status["shortage"]`
    5. `sample.stock -= order.quantity`
    6. `order.transition_to(OrderStatus.CONFIRMED)`
    7. `self._production_service.dequeue()`
