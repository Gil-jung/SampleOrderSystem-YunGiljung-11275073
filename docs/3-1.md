# [Phase 3] 사이클 3-1 — 주문 예약 생성

`PLAN.md`의 Phase 3 [3-1] 항목에 대한 tdd 스킬 사이클 기록.

## RED

### 목표 (단일 동작)

`OrderService.reserve(sample_id, customer_name, quantity)`를 호출하면 `RESERVED` 상태의
주문이 생성되어 `OrderRepository`에 저장된다. (PRD.md 3.1~3.2절)

### 설계 메모

- `Order`는 [1-3]에서 `order_id` 없이 설계했다(YAGNI). 주문 식별자는 [2-1]의 `Sample`과
  달리 사용자가 주지 않으므로, `OrderRepository.add(order)`가 저장 시 `order_id`를
  발급하고 반환한다 (예: `"ORD-0001"`부터 순번 발급). `OrderService.reserve`도 이 `order_id`를
  그대로 반환해 호출자가 이후 승인/거절 등에서 참조할 수 있게 한다.
- [2-1]과 동일한 패턴으로, `OrderRepository`는 이번 사이클에서 실제 협력 객체로 함께
  도입하고 별도 단위 테스트는 만들지 않는다.
- 이번 사이클에서는 `sample_id`가 실제로 등록된 시료인지 검증하지 않는다 ([3-2]에서 다룬다).

### 작성할 실패 테스트

`tests/service/test_order_service.py` (신규 파일)

```python
from model.order import OrderStatus
from repository.order_repository import OrderRepository
from service.order_service import OrderService


def test_주문_예약_생성_시_RESERVED_상태_주문이_저장된다():
    order_repository = OrderRepository()
    service = OrderService(order_repository)

    order_id = service.reserve(sample_id="SMP-001", customer_name="홍길동", quantity=5)

    stored = order_repository.get(order_id)
    assert stored.sample_id == "SMP-001"
    assert stored.customer_name == "홍길동"
    assert stored.quantity == 5
    assert stored.status == OrderStatus.RESERVED
```

### 구현 방향 (GREEN 예고 — RED 단계에서는 작성하지 않음)

- `src/repository/order_repository.py`: `OrderRepository`
  - `add(order) -> str`: 순번 기반 `order_id`(`"ORD-0001"`, `"ORD-0002"`, ...)를 발급해
    `dict`에 저장하고 발급한 `order_id`를 반환.
  - `get(order_id) -> Order`: 저장된 주문 반환.
- `src/service/order_service.py`: `OrderService`
  - `__init__(self, order_repository)`
  - `reserve(sample_id, customer_name, quantity) -> str`: `Order`를 생성해
    `order_repository.add()`로 저장하고 발급된 `order_id`를 반환
    (`Order`의 기본값으로 초기 상태 `RESERVED`가 이미 보장됨 — [1-3]).
