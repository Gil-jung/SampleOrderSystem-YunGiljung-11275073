# 데이터 모니터링 Tool PoC

## 1. 개요

### 1.1 목적
운영 중인 애플리케이션의 메모리(인메모리 자료구조)에 저장된 데이터 상태를 별도의 DB 접속이나
디버거 없이, **콘솔에서 실시간으로 조회**할 수 있는 경량 관리자 도구를 검증한다.
본 PoC에서는 주문관리 시스템을 예시 도메인으로 삼아 기능을 구현하고 테스트한다.

### 1.2 범위
- 대상 데이터: 프로세스 내 Python `dict` / `list` 기반 인메모리 저장소
- 조회 방식: 일정 주기(polling)로 콘솔 화면을 자동 갱신하는 watch 형태
- 검증 시나리오: 주문관리 시스템(주문 생성/상태 변경/취소)
- 산출물: 데이터 모니터링 Tool(PoC) + 시나리오 기반 테스트 결과 + 개선 후 최종 구현

### 1.3 비범위 (Out of Scope)
- 실제 RDB/NoSQL 연동 (PostgreSQL, MongoDB 등)
- 다중 프로세스/분산 환경에서의 데이터 수집
- 인증/인가, 원격 접속(SSH 등)을 통한 모니터링
- 데이터 변경(쓰기) 기능 — 본 도구는 조회 전용(read-only)

## 2. Tool 설계

### 2.1 핵심 기능
| 기능 | 설명 |
|---|---|
| 실시간 조회 | 지정한 주기(기본 1~2초)마다 콘솔 화면을 지우고 최신 데이터 상태를 다시 그림 |
| 테이블 뷰 | 저장소 내 레코드를 표 형태로 정렬하여 가독성 있게 출력 |
| 요약 통계 | 전체 건수, 상태별 건수 등 요약 정보를 상단에 표시 |
| 필터/정렬 옵션 | 특정 상태만 보기, 특정 컬럼 기준 정렬 (CLI 옵션으로 지정) |
| 변경 하이라이트 | 직전 스냅샷과 비교하여 변경된 레코드를 표시 (신규/상태변경) |
| 안전한 조회 | 원본 데이터에 락(lock) 없이 접근 시에도 예외가 발생하지 않도록 스냅샷 복사 후 렌더링 |

### 2.2 아키텍처

```
+------------------+       +--------------------+       +------------------+
|  Order Service    | ----> |  In-Memory Store    | <---- |  Data Monitor    |
|  (주문 생성/변경)  |       |  (dict/list)        |       |  Tool (Console)  |
+------------------+       +--------------------+       +------------------+
                                                                 |
                                                                 v
                                                        주기적 polling
                                                        + 콘솔 렌더링
```

- **In-Memory Store**: 주문관리 시스템이 사용하는 데이터 저장소. Tool과 동일 프로세스 내 객체를
  참조하거나, 데모 목적으로는 별도 모듈에서 공유 객체로 노출한다.
- **Data Monitor Tool**: 저장소 참조를 받아 `interval` 초마다 스냅샷을 떠서 콘솔에 출력하는
  독립 실행 모듈. 저장소 구현을 몰라도 되도록 최소 인터페이스(`snapshot() -> list[dict]`)만 의존한다.

### 2.3 최소 인터페이스 정의
```python
class Monitorable(Protocol):
    def snapshot(self) -> list[dict]:
        """현재 데이터 상태의 얕은 복사본 리스트를 반환한다."""
```
- 어떤 저장소든 `snapshot()`만 구현하면 Tool에 연결 가능 → 주문관리 외 다른 도메인에도 재사용 가능한 구조.

### 2.4 CLI 사용 예시
```bash
python monitor.py --interval 1.5 --filter status=PAID --sort created_at
```

## 3. 검증 시나리오: 주문관리 시스템

### 3.1 데이터 모델
```python
Order = {
    "order_id": str,
    "customer": str,
    "items": list[str],
    "amount": int,
    "status": str,   # CREATED -> PAID -> SHIPPING -> DELIVERED / CANCELLED
    "created_at": str,
    "updated_at": str,
}
```

### 3.2 테스트 시나리오 목록

| ID | 시나리오 | 검증 항목 |
|---|---|---|
| S1 | 주문 신규 생성 | Tool 화면에 신규 레코드가 다음 polling 주기 내 반영되는가 |
| S2 | 주문 상태 변경 (CREATED→PAID→SHIPPING→DELIVERED) | 상태 변경이 지연 없이 반영되고, 변경 레코드가 하이라이트되는가 |
| S3 | 주문 취소 (CANCELLED) | 취소 상태 및 사유가 정확히 표시되는가 |
| S4 | 대량 동시 생성 (예: 1,000건 동시 insert) | 렌더링 지연/누락 없이 전체 건수가 정확히 집계되는가 |
| S5 | 조회 중 데이터 변경 (race condition) | 스냅샷 렌더링 중 원본 데이터가 변경되어도 Tool이 예외 없이 동작하는가 |
| S6 | 필터/정렬 옵션 적용 | `status=PAID` 필터, `created_at` 정렬이 올바르게 동작하는가 |
| S7 | 빈 저장소 상태 | 데이터가 0건일 때 에러 없이 "데이터 없음" 메시지를 표시하는가 |
| S8 | 장시간 실행 안정성 | 수 시간 연속 polling 시 메모리 누수나 CPU 사용량 급증이 없는가 |

### 3.3 테스트 방법
1. 주문관리 시나리오 스크립트(`order_simulator.py`)로 주문 생성/상태 변경/취소 이벤트를 일정 간격으로 발생시킨다.
2. Data Monitor Tool을 별도 스레드 또는 별도 프로세스에서 기동하여 동일 저장소를 관찰한다.
3. 각 시나리오(S1~S8)를 순서대로 실행하며 콘솔 출력을 스크린샷/로그로 기록한다.
4. `pytest` 기반 단위 테스트로 `snapshot()` 반환값의 정합성(건수, 필드 무결성)을 자동 검증한다.

### 3.4 결과 판정 기준
- 모든 시나리오(S1~S8)에서 **데이터 유실 없음**, **예외 미발생**, **표시 지연이 설정 주기의 2배 이내**일 것.
